# AGENT.md

이 레포는 크래프톤 정글 캠퍼스 생활 안내 AI FAQ 게시판을 만들면서 학습하기 위한 프로젝트다.

## 프로젝트 방향

최종 목표는 사람들이 캠퍼스 생활 관련 질문을 검색하고, FAQ 게시글을 올리고, AI 에이전트가 RAG를 이용해 관련 답변을 댓글로 달아주는 게시판이다.

캠퍼스 생활 안내의 원천 reference는 크래프톤 정글 Notion 페이지다.

- URL: https://kraftonjungle.notion.site/junglecampuslife
- 이 페이지는 향후 FAQ 데이터, 검색, RAG 답변 근거의 주요 reference로 사용한다.
- Notion 원문은 바뀔 수 있으므로, 실제 RAG 데이터로 쓰기 전에는 별도 스냅샷 또는 추출본을 만들어 버전을 남긴다.

하지만 한 번에 전체 기능을 만들지 않는다. 매번 아주 작은 단위로 구현하고, 이해하고, 주석을 정리하고, 커밋한 뒤 다음 단계로 넘어간다.

## 작업 원칙

1. 큰 기능을 한 번에 만들지 않는다.
2. 매 작업은 하나의 작고 검증 가능한 단위로 제한한다.
3. 코드 작성 전에 반드시 다음을 먼저 설명한다.
   - 지금 하려는 것
   - 왜 필요한지
   - 관련 개념
   - 이번 단계에서 만들 파일 또는 수정할 파일
4. 코드 작성 후에는 반드시 다음을 정리한다.
   - 무엇이 바뀌었는지
   - 실행 또는 테스트 방법
   - 이번 단계에서 이해해야 할 핵심 개념
5. 한 단위가 끝나면 커밋한다.
6. 다음 단위로 넘어가기 전에 사용자가 이해했는지 확인한다.

## 학습 방식

사용자는 React, FastAPI, PostgreSQL, SQLAlchemy, AI 기능을 프로젝트를 만들면서 배우는 중이다.

설명은 코드보다 개념과 배경을 우선한다. 단순히 "이 코드를 써라"가 아니라, 왜 이 코드가 필요한지 설명해야 한다.

권장 설명 순서:

1. 개념적 역할
2. 왜 필요한지
3. 현재 프로젝트에서 어디에 쓰이는지
4. 최소 코드
5. 실행 결과
6. 사용자가 직접 말로 설명해볼 체크 질문

## 코드 주석 원칙

코드는 사용자가 나중에 다시 봐도 흐름을 이해할 수 있어야 한다.

1. 새로 작성하는 주요 함수에는 함수 단위 설명 주석을 단다.
2. 함수 설명 주석은 목적, 입력, 출력, 사용 맥락 중심으로 작성한다.
3. 함수 설명 주석은 최대 5줄 이내로 유지한다.
4. 핵심 로직에는 짧은 인라인 주석을 단다.
5. 인라인 주석은 "왜 이 처리가 필요한지"를 설명하고, 코드가 그대로 말하는 내용은 반복하지 않는다.
6. 너무 당연한 문법 설명 주석은 피한다.
7. 새로 추가하는 구조 파일이나 설정 파일에도 존재 이유를 짧게 남긴다.
   - 예: `__init__.py`, `requirements.txt`, `package.json`, `vite.config.ts`, `docker-compose.yml`
8. 라이브러리나 도구가 처음 등장하면 그 역할을 한 줄로 설명한다.
   - 예: `uvicorn`은 FastAPI 앱을 실제 HTTP 서버로 실행하는 프로그램
9. 커밋 전에 새 파일을 훑고, 사용자가 나중에 봤을 때 "왜 필요한 파일인지" 알 수 있는지 확인한다.

## 금지 사항

1. 사용자가 명시적으로 명령하거나 허락하기 전까지 파일이나 폴더를 삭제하지 않는다.
2. 사용자가 요청하지 않은 대규모 리팩터링을 하지 않는다.
3. 폴더 구조와 아키텍처 분리는 나중에 진행하되, 먼저 전체 흐름을 이해할 수 있게 한다.
4. React, FastAPI, DB, AI 기능을 한 번에 붙이지 않는다.
5. 학습 목적의 주석을 무분별하게 많이 달지 않는다. 필요한 곳에만 짧게 단다.
6. 사용자가 혼란스러워하면 구현을 멈추고 개념 설명으로 돌아간다.

## 우선 구현 순서

1. 프로젝트 기본 뼈대
2. FastAPI `GET /health`
3. FastAPI `GET /posts` 정적 데이터
4. React 프로젝트 기본 뼈대
5. React에서 `GET /posts` 호출
6. PostgreSQL 연결
7. SQLAlchemy 모델 도입
8. React 화면에 DB 게시글 목록 표시
9. `POST /posts` 글 작성
10. 검색
11. 비슷한 FAQ 추천 설계
12. 댓글
13. RAG 기반 AI 댓글 POC

각 단계는 독립적으로 이해, 실행, 커밋 가능해야 한다.

## 커밋 원칙

커밋은 작게 만든다.

예시:

- `chore: add project workflow rules`
- `feat: add FastAPI health endpoint`
- `feat: add static posts endpoint`
- `feat: scaffold React app`
- `feat: render posts from API`

커밋 전에 테스트 또는 실행 확인을 하고, 확인하지 못했다면 이유를 명확히 남긴다.

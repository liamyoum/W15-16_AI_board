# 나만무 프로젝트 주제 선정 가이드

이 문서는 동석 코치님의 프로젝트 주제 선정 코멘트를 바탕으로, 나중에 팀 프로젝트 아이디어를 정할 때 참고하기 위한 기준을 정리한 문서입니다.

## 핵심 방향

프로젝트는 개발자로서 기술적으로 어필할 수 있는 주제여야 한다.

단순히 기능이 많은 서비스보다, 인터뷰에서 "이 프로젝트를 하면서 어떤 기술적 문제를 풀었는가"를 구체적으로 이야기할 수 있는 주제가 좋다. 사용자가 있는 서비스여야 하지만, 고객 관점만 앞세워 기술적 깊이가 부족해지면 프로젝트의 설득력이 약해진다.

좋은 프로젝트의 기준:

- 기술 중심의 주제다.
- 인터뷰에서 기술적으로 깊게 설명할 수 있다.
- 데모가 가능하다.
- 고객과 서비스가 명확하게 연결되어 있다.
- 5-6명이 5주 동안 밀도 있게 파고들 수 있는 scope이다.
- 기능의 양보다 기술적 깊이와 완성도를 높일 수 있다.

## Scope 기준

많은 프로젝트가 reject되는 이유는 scope이 너무 작기 때문이다.

미니 프로젝트를 4명이 3박 4일 동안 진행했다면, 나만무 프로젝트는 5-6명이 5주 동안 진행하는 만큼 그에 맞는 기술적 깊이가 있어야 한다. 단순히 화면이나 기능 개수를 늘리라는 뜻이 아니다. 같은 주제라도 성능, 안정성, 확장성, 동시성, 데이터 처리, 운영 자동화, 사용자 경험 완성도 등을 깊게 파고들어야 한다.

적절한 scope의 방향:

- 기능 수를 무리하게 늘리지 않는다.
- 핵심 기술 챌린지를 분명히 잡는다.
- 실제 서비스처럼 완성도를 높인다.
- 부하 테스트, 장애 상황, 데이터 증가, 동시 사용자 상황을 고려한다.
- 팀원 모두가 기술적으로 설명할 수 있는 부분을 가진다.

피해야 할 scope:

- CRUD만 있는 작은 서비스.
- 발표에서 기술적으로 말할 내용이 적은 서비스.
- 기능은 많지만 내부적으로 단순한 서비스.
- 외부 API 조합에 대부분 의존하는 서비스.
- 데모가 불안정하거나 현장에서 보여주기 어려운 서비스.

## Demo 가능성

발표가 있으므로 반드시 Demo를 할 수 있는 아이디어여야 한다.

아이디어가 좋아도 발표장에서 안정적으로 보여줄 수 없으면 리스크가 크다. 특히 환경 의존도가 높거나 정확도 문제가 큰 기능은 주의해야 한다.

주의할 아이디어:

- GPS 기반 기능: 실내 환경에서는 정확도가 떨어질 수 있다.
- 네트워크 양방향 통신이 핵심인 기능: 데모 환경에서 연결 문제나 지연 문제가 발생할 수 있다.
- 외부 장비, 위치, 특정 환경에 강하게 의존하는 기능.
- 현장에서 실패했을 때 대체 시나리오가 없는 기능.

Demo 관점 체크:

- [ ] 발표장에서 핵심 기능을 시간 내에 보여줄 수 있는가?
- [ ] 네트워크가 불안정해도 최소한의 시연이 가능한가?
- [ ] 더미 데이터나 시뮬레이션으로 핵심 기술을 보여줄 수 있는가?
- [ ] 실패 상황을 대비한 백업 시나리오가 있는가?
- [ ] 사용자가 어떤 문제를 겪고, 서비스가 어떻게 해결하는지 한눈에 보이는가?

## 바람직한 프로젝트 방향

### 1. Traffic을 많이 실어 테스트할 수 있는 서비스

부하를 걸어볼 수 있는 서비스는 기술적으로 이야기할 거리가 많다.

예시 관점:

- 동시 요청 처리.
- 대기열 처리.
- 캐싱 전략.
- DB index와 query optimization.
- rate limit.
- 실시간 상태 업데이트.
- 장애 상황에서의 복구.

이런 주제는 결과물 자체뿐 아니라 "왜 이런 병목이 생겼고, 어떻게 측정하고 개선했는가"를 발표와 인터뷰에서 설명하기 좋다.

### 2. 기술적으로 challenging한 서비스 모방

이미 존재하는 서비스 중 기술적으로 어려운 부분을 모방하고, 기존 서비스의 단점을 개선하는 방향도 좋다.

중요한 점은 단순 클론이 아니라, reference를 조사하고 그 서비스가 해결하는 기술 문제를 이해하는 것이다.

좋은 접근:

- 기존 서비스를 조사한다.
- 핵심 기술 문제를 뽑는다.
- 우리가 개선할 단점이나 차별점을 정의한다.
- 규모를 줄이더라도 기술적 본질은 유지한다.

예시:

- 실시간 협업 도구의 동시 편집 문제.
- 대기열 기반 예약 서비스의 트래픽 집중 문제.
- 검색 서비스의 랭킹과 인덱싱 문제.
- 알림 서비스의 이벤트 처리와 전송 보장 문제.
- 개발자 도구의 로그, 모니터링, 자동화 문제.

### 3. Customer와 서비스가 paired 되어 있는 서비스

customer가 반드시 일반 소비자일 필요는 없다. 개발자가 customer여도 되고, B2B 서비스도 가능하다.

중요한 것은 고객 페르소나와 서비스 방향이 논리적으로 맞아야 한다는 점이다.

나쁜 예:

- 노인을 위한 앱인데 UI 폰트가 10pt인 경우.
- 개발자를 위한 도구인데 로그나 디버깅 경험이 부족한 경우.
- B2B 운영 도구인데 권한 관리나 감사 로그가 없는 경우.

좋은 접근:

- 고객을 구체적으로 정의한다.
- 고객이 겪는 문제를 명확히 쓴다.
- 그 문제를 해결하기 위해 필요한 기술을 연결한다.
- UI, 기능, 성능, 안정성 기준을 고객에 맞춘다.

고객 예시:

- 개발자.
- 팀 리더.
- CS 담당자.
- 운영자.
- 채용 담당자.
- 온라인 강의 운영자.
- 커뮤니티 관리자.
- 소규모 B2B 팀.

## 기술 챌린지와 시장성의 중간점

좋은 주제는 기술적으로 어렵기만 해도 안 되고, 시장성만 있어도 부족하다.

기술적인 챌린지와 고객 문제 사이의 연결이 분명해야 한다.

검토 질문:

- 이 서비스의 고객은 누구인가?
- 고객은 어떤 문제 때문에 이 서비스를 쓰는가?
- 그 문제를 해결하려면 어떤 기술적 난제가 있는가?
- 우리가 그 난제를 직접 구현하거나 검증할 수 있는가?
- 5주 안에 데모 가능한 수준으로 만들 수 있는가?
- 발표에서 기술적 깊이를 보여줄 수 있는가?

## 서비스 방향성 확인법

### 1. 뉴스 기사 3-5줄 작성하기

우리가 만든 제품을 뉴스 기사처럼 3-5줄로 소개해본다.

이 짧은 소개 안에 아래 내용이 모두 들어가야 한다.

- 누가 쓰는 서비스인지.
- 어떤 문제를 해결하는지.
- 어떤 방식으로 해결하는지.
- 왜 지금 이 서비스가 의미 있는지.
- 기술적으로 무엇이 핵심인지.

예시 형식:

```text
OO팀은 개발자를 위한 실시간 장애 재현 게시판 서비스를 공개했다.
이 서비스는 사용자가 에러 로그와 재현 절차를 등록하면 AI가 원인을 요약하고 유사 사례를 찾아준다.
대규모 로그 검색과 실시간 댓글 협업을 지원해 팀 단위 디버깅 시간을 줄이는 것이 목표다.
팀은 Redis 캐싱, 비동기 작업 큐, 검색 인덱싱을 활용해 반복 조회와 AI 분석 비용을 최적화했다.
```

이 3-5줄을 읽었을 때 customer, product, value, technical challenge가 선명하지 않다면 방향을 다시 잡아야 한다.

### 2. FAQ 미리 작성하기

사용자가 할 법한 질문을 먼저 list-up하고 답변을 작성한다.

FAQ를 미리 작성하면 제품의 경계가 선명해진다. 어떤 기능을 넣어야 하는지, 어떤 기능은 필요 없는지 판단하기 쉬워진다.

FAQ 예시:

- 이 서비스는 누구를 위한 것인가?
- 기존 서비스와 무엇이 다른가?
- 사용자는 어떤 상황에서 이 서비스를 쓰는가?
- 가장 중요한 기능은 무엇인가?
- AI 기능은 왜 필요한가?
- AI가 틀렸을 때 사용자는 어떻게 확인하거나 수정하는가?
- 데이터가 많아지면 어떻게 처리하는가?
- 동시에 사용자가 몰리면 어떻게 되는가?
- 데모에서는 어떤 시나리오를 보여줄 것인가?

## DevOps와 Data Engineering 주제

DevOps나 Data Engineering 중심으로 가고 싶다면 아이디어를 만들기 어려울 수 있다. 이런 방향이 있다면 코치님과 먼저 상의하는 것이 좋다.

가능한 방향:

- Kubernetes와 유사한 기능을 작게 구현해보기.
- 배포 자동화 플랫폼.
- 로그 수집과 분석 시스템.
- 데이터 파이프라인 모니터링 도구.
- feature flag 또는 canary 배포 도구.
- queue 기반 작업 처리 플랫폼.

이런 주제는 일반 사용자 서비스보다 기술 설명이 강점이 될 수 있다. 대신 customer와 demo 시나리오를 더 명확히 잡아야 한다.

## 분업 방식

기능 단위로 나누되, 모두가 풀스택을 경험할 수 있게 나누는 방향이 좋다.

피해야 할 분업:

- 한 명은 프론트엔드만, 한 명은 백엔드만, 한 명은 배포만 담당하는 식의 고정 분업.
- 특정 팀원만 핵심 기술을 이해하는 구조.
- 발표 때 일부 팀원만 기술 설명을 할 수 있는 구조.

권장 분업:

- 기능 단위 ownership을 나눈다.
- 각 기능 담당자가 frontend, backend, database, test를 함께 본다.
- 공통 인프라와 아키텍처는 팀 전체가 이해한다.
- 각자 맡은 기능에 기술적 챌린지를 하나씩 포함한다.

예시:

- A: 인증과 권한, OAuth, session/JWT.
- B: 게시글/댓글 도메인, ERD, transaction.
- C: 검색과 캐싱, Redis, query optimization.
- D: AI 분석 기능, Job Queue, SSE.
- E: 배포, 로깅, 부하 테스트, 모니터링.

- [ ] 고객이 명확한가?
- [ ] 고객의 문제가 구체적인가?

- [ ] 3-5줄 뉴스 기사로 설명했을 때 설득력이 있는가?
- [ ] FAQ를 작성했을 때 제품 경계가 선명한가?

기술:

- [ ] 인터뷰에서 기술적으로 이야기할 핵심 주제가 있는가?
- [ ] 단순 CRUD를 넘어서는 챌린지가 있는가?
- [ ] 트래픽, 동시성, 캐싱, 비동기 처리, 검색, 실시간성, 배포 중 최소 하나 이상의 깊이가 있는가?
- [ ] reference를 조사할 수 있는 기존 서비스나 기술 사례가 있는가?
- [ ] 팀원 모두가 기술적으로 설명할 부분을 가질 수 있는가?

Scope:

- [ ] 5-6명이 5주 동안 밀도 있게 할 만한 크기인가?
- [ ] 기능 수가 아니라 기술적 완성도를 높이는 방향인가?
- [ ] 과하게 크거나 데모 불가능한 부분은 없는가?
- [ ] 핵심 기능과 선택 기능이 구분되어 있는가?

Demo:

- [ ] 발표장에서 안정적으로 보여줄 수 있는가?
- [ ] 핵심 사용자 시나리오가 명확한가?
- [ ] 실패했을 때 대체 가능한 시나리오가 있는가?
- [ ] 기술적 챌린지가 화면이나 수치로 드러나는가?

## 최종 판단 기준

좋은 주제는 아래 문장에 답할 수 있어야 한다.

```text
우리는 [고객]이 겪는 [문제]를 해결하기 위해 [서비스]를 만들고,
그 과정에서 [기술적 챌린지]를 [구체적인 방식]으로 풀어낸다.
발표에서는 [데모 시나리오]를 통해 이 문제 해결 과정을 보여준다.
```

이 문장이 자연스럽게 완성되지 않으면 주제를 더 다듬어야 한다.
